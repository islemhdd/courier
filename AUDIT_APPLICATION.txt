1. ARCHITECTURE GÉNÉRALE
══════════════════════════════════════════════════════════════════════════════

Architecture: Monolithe avec API REST + Frontend SPA
Type: Application de gestion de courrier administratif
Backend: Laravel 12.x (PHP 8.x)
Frontend: React 19 avec Vite (SPA dans /resources/js/mon-projet)
Base de données: SQL (via Eloquent ORM)
Authentification: Laravel Sanctum (API tokens)
Temps réel: Laravel Echo + Pusher (notifications broadcast)
Rendu: Côté client uniquement (pas de SSR Laravel/Inertia)

Points forts:
- Séparation claire entre API et Frontend
- Architecture de permissions granulaire (rôles, scopes)
- Cache côté client avec invalidation
- Notifications temps réel via WebSockets

Chronologie:
  date_creation         Date de création
  date_reception        Date de réception
  date_limite_reponse   Deadline calculée
  repondu_le            Timestamp réponse
  transmis_le           Timestamp transmission

Acteurs:
  expediteur            Texte libre (externe)
  destinataire          Texte libre (externe)
  source_id             FK → sources
  createur_id           FK → users
  valideur_id           FK → users
  transmis_par_id       FK → users

Diffusion:
  mode_diffusion        unicast | multicast | broadcast
  service_source_id     FK → services
  service_destinataire_id  FK → services
  structure_origine_id  FK → structures
  structure_destinataire_id FK → structures

Workflow:
  statut                CREE | NON_VALIDE | VALIDE | TRANSMIS | RECU
  requiert_reponse      boolean
  delai_reponse_jours   integer
  transmission_demandee boolean
  parent_courrier_id    FK → courriers (réponse à)
  validation_parent_id  FK → courriers
  niveau_confidentialite_id FK → niveau_confidentialites

Relations clés:
  Courrier 1──N CourrierRecipient (destinataires de diffusion)
  Courrier 1──N CourrierPerson (personnes concernées)
  Courrier 1──N CourrierAttachment (pièces jointes)
  Courrier 1──N CourrierComment (commentaires/instructions)
  Courrier 1──N Message (messages liés)
  Courrier 1──N Courrier (réponses = parent_courrier_id)

5. FLUX DE TRAVAIL (WORKFLOW)
══════════════════════════════════════════════════════════════════════════════

Création d'un courrier entrant:
  1. Secretaria général remplit le formulaire
  2. Définit mode_diffusion (unicast/multicast/broadcast)
  3. Définit recipients (structure/service/user)
  4. Statut initial: RECU
  5. Notification envoyée aux destinataires
  6. Apparait dans "Courriers reçus" des destinataires

Création d'un courrier sortant:
  1. Chef général/secrétaire remplit le formulaire
  2. Définit source (expéditeur externe)
  3. Peut être une réponse à un courrier existant
  4. Statut: VALIDE (si chef général) ou CREE (sinon)
  5. Apparait dans "Courriers envoyés" du créateur
  6. Pas de destinataires internes (diffusion externe)

Cycle de validation:
  1. Courrier en statut CREE
  2. Chef valide → statut VALIDE
  3. Chef rejette → statut NON_VALIDE

Transmission:
  1. Courrier en statut VALIDE, RECU ou TRANSMIS
  2. Expéditeur définit les nouveaux destinataires
  3. Statut → TRANSMIS
  4. Notification aux nouveaux destinataires
  5. Apparait dans "Courriers reçus" des destinataires

Réponse:
  1. Courrier avec requiert_reponse=true
  2. Destinataire clique "Répondre"
  3. Formulaire simplifié (ReplyCourrierForm)
  4. Destinataires automatically définis (getReplyRecipients)
  5. Courrier créé avec parent_courrier_id
  6. Notification envoyée aux concernés par le parent
  7. Parent marqué repondu_le = now()

Archivage:
  1. Archive créée avec copie des données
  2. Courrier original supprimé
  3. Archive visible dans la page "Archives"

6. CORRECTIONS APPLIQUÉES
══════════════════════════════════════════════════════════════════════════════

CORRECTION #1: Réponses invisibles dans liste des reçus (Problèmes 2 & 4)
─────────────────────────────────────────────────────────────────────────

Fichier:  app/Http/Controllers/CourrierController.php
Sévérité: CRITIQUE

Cause:
  - Les réponses étaient forcées en TYPE_SORTANT
  - L'endpoint /courriers/recus filtre par type=entrant
  - syncRecipients() ignorait les courriers TYPE_SORTANT
  → Les réponses n'apparaissaient nulle part pour les destinataires

Correctif:
  1. Changé le type des réponses de TYPE_SORTANT vers TYPE_ENTRANT
  2. syncRecipients() ne skip plus les TYPE_SORTANT si parent_courrier_id existe
  3. La méthode envoyes() inclut désormais les courriers TYPE_ENTRANT
     dont l'utilisateur est le créateur (via: WHERE type=sortant OR createur_id=X)

Impact:
  - Les réponses apparaissent maintenant dans "Courriers reçus" des destinataires
  - Les créateurs de réponses les voient dans "Courriers envoyés"
  - Les notifications de réponse pointent vers la page de détail correcte

───

CORRECTION #2: Dancing list (Problème 3)
─────────────────────────────────────────

Fichiers: resources/js/mon-projet/src/pages/SentCourriers.jsx
          resources/js/mon-projet/src/pages/Messages.jsx
Sévérité: HAUTE

Cause:
  - selectedCourrier?.id dans le tableau de dépendances de loadData()
  - applyCourriers() modifie selectedCourrier
  → Changement de sélection → recréation de loadData → ré-exécution de
     l'useEffect → nouveau chargement → changement de sélection → boucle

Correctif:
  - Remplacé selectedCourrier?.id par un useRef (selectedIdRef)
  - La sélection ne déclenche plus de rechargement intempestif

Impact:
  - Plus de "saut" de sélection lors des rechargements
  - Stabilité de l'interface utilisateur

───

CORRECTION #3: Notification message non ouvert (Problème 6)
─────────────────────────────────────────────────────────────

Fichier:  resources/js/mon-projet/src/pages/Messages.jsx
Sévérité: MOYENNE

Cause:
  - getNotificationRoute() renvoie /messages?messageId=X
  - Messages.jsx n'analysait pas le query parameter messageId
  → Le clic sur notification n'ouvrait aucun message spécifique

Correctif:
  - Ajout de useSearchParams pour détecter messageId dans l'URL
  - Si un messageId est présent et que le message est dans la liste, 
    il est automatiquement sélectionné
  - Sinon, une requête API fetch le message directement
  - Gestion du mailbox (recu/envoye) automatique

Impact:
  - Le clic sur une notification de message ouvre le bon message

7. PROBLÈMES RESTANTS / OBSERVATIONS
══════════════════════════════════════════════════════════════════════════════

OBSERVATION #1 - Build (Pré-existant)
─────────────────────────────────────
- Le fichier vite.config.js à la racine référence resources/js/app.tsx
- Ce fichier n'existe pas (l'app est dans mon-projet/)
- Le build root échoue, mais mon-projet a son propre build qui fonctionne
- Suggéré: aligner la config racine sur mon-projet ou supprimer la config racine

OBSERVATION #2 - Type filter values
─────────────────────────────────────
- Les filtres côté client utilisent 'arrivee', 'depart', 'interne'
- Les données en base utilisent 'entrant', 'sortant'
- Le filtre type dans ReceivedCourriers ne correspond pas aux données réelles
- Analyse: le filtre client type = 'arrivee' est passé à l'API qui attend 'entrant'
  → Le filtre ne fonctionne pas correctement (toujours 0 résultats si ce filtre est actif)
- Suggéré: aligner les valeurs du filtre sur 'entrant'/'sortant' OU les mapper dans l'API

OBSERVATION #3 - Cache pagination
─────────────────────────────────────
- Le cache client (pageCache.js) stocke les résultats de pagination
- Si la page 2 est en cache et que la page 1 est modifiée, la page 2
  peut montrer des résultats obsolètes
- Faible priorité

OBSERVATION #4 - Aucune gestion d'erreur réseau
─────────────────────────────────────
- Les appels API utilisent .catch() génériques
- Pas de retry automatique
- Pas de fallback offline

OBSERVATION #5 - Sécurité
─────────────────────────────────────
- L'application utilise des permissions côté backend (via les méthodes peutEtre*
  sur le modèle Courrier)
- Les tokens Sanctum expirent et sont gérés correctement
- Les politiques (Policies) ne sont pas utilisées (tout est dans le modèle)
- C'est acceptable mais les Policies apporteraient une meilleure séparation
