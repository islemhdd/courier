================================================================================
  DOCUMENTATION : GESTION DES UTILISATEURS
  Module de gestion des comptes, rôles et permissions
  Application Gestion de Courrier
================================================================================


1. PRESENTATION
--------------------------------------------------------------------------------
Le module de gestion des utilisateurs permet de créer, modifier, consulter
et supprimer les comptes utilisateurs de l'application. Il implémente un
système de contrôle d'accès basé sur les rôles (RBAC) avec une hiérarchie
organisationnelle à deux niveaux (Structure > Service).

Les utilisateurs sont authentifiés via Laravel Sanctum (tokens d'API) et
leurs permissions sont vérifiées à chaque action via :
  - Les Policies Laravel (UserPolicy.php)
  - Les méthodes métier sur le modèle User (User.php)
  - Les contrôles dans le contrôleur (UserController.php)
  - Les contrôles frontend (Users.jsx)


2. STRUCTURE DE LA TABLE "users"
--------------------------------------------------------------------------------
La table users contient les colonnes suivantes :

  Champ                     Type          Description
  ------------------------- ------------- ------------------------------------
  id                        bigint        Identifiant unique (auto-incrément)
  nom                       varchar(255)  Nom de famille
  prenom                    varchar(255)  Prénom
  email                     varchar(255)  Adresse email (unique)
  password                  varchar(255)  Mot de passe (hashé avec Bcrypt)
  actif                     boolean       Compte actif ou désactivé
  role                      varchar(50)   Rôle : admin | chef | secretaire
  role_scope                varchar(50)   Périmètre : general | structure |
                                          service
  service_id                bigint|null   Référence vers la table services
  structure_id              bigint|null   Référence vers la table structures
  niveau_confidentialite_id bigint|null   Référence vers la table
                                          niveau_confidentialites
  remember_token            varchar(100)  Token de session "Se souvenir de moi"
  email_verified_at         timestamp     Date de vérification email
  created_at                timestamp     Date de création
  updated_at                timestamp     Date de modification


3. MODELES DE DONNEES LIES
--------------------------------------------------------------------------------

3.1. Table "structures" (directions / entités organisationnelles)
  Champ               Type          Description
  ------------------- ------------- -----------------------------------------
  id                  bigint        Identifiant unique
  libelle             varchar(255)  Nom de la structure (ex: "Direction des
                                    Ressources Humaines")
  chef_structure_id   bigint|null   Référence vers l'utilisateur chef de
                                    cette structure
  created_at          timestamp
  updated_at          timestamp

3.2. Table "services" (départements / divisions)
  Champ               Type          Description
  ------------------- ------------- -----------------------------------------
  id                  bigint        Identifiant unique
  libelle             varchar(255)  Nom du service (ex: "Service Paie")
  structure_id        bigint|null   Référence vers la structure parente
  created_at          timestamp
  updated_at          timestamp

  Relation : Un service appartient à UNE structure.
             Une structure peut avoir PLUSIEURS services.

3.3. Table "niveau_confidentialites"
  Champ               Type          Description
  ------------------- ------------- -----------------------------------------
  id                  bigint        Identifiant unique
  libelle             varchar(255)  Libellé (ex: "Public", "Interne",
                                    "Confidentiel", "Secret")
  rang                integer       Ordre hiérarchique (plus le chiffre est
                                    élevé, plus le niveau est restrictif)
  created_at          timestamp
  updated_at          timestamp


4. ROLES ET PERIMETRES
--------------------------------------------------------------------------------

4.1. Les 3 rôles (User::ROLES)

  Rôle "admin" (ROLE_ADMIN)
  --------------------------
  - Accès complet à toutes les fonctionnalités
  - Voit tous les courriers, tous les utilisateurs
  - Peut créer des courriers entrants
  - Peut valider, transmettre, archiver, supprimer n'importe quel courrier
  - Gère tous les utilisateurs (création, modification, suppression)
  - Scope automatiquement forcé à "general"
  - Constant : User::ROLE_ADMIN = 'admin'

  Rôle "chef" (ROLE_CHEF)
  ------------------------
  - Gère et valide le courrier de son périmètre
  - Peut créer des courriers (sauf entrants, réservé au chef général)
  - Valide les courriers de son service/structure/général
  - Transmet les courriers validés
  - Gère les utilisateurs de son périmètre
  - Scope : general, structure ou service
  - Constant : User::ROLE_CHEF = 'chef'

  Rôle "secretaire" (ROLE_SECRETAIRE)
  ------------------------------------
  - Crée et suit le courrier
  - Peut créer des courriers sortants
  - Ne peut PAS valider directement
  - Peut initier une transmission (doit être validée par un chef)
  - Peut archiver les courriers de son service
  - Ne gère PAS les utilisateurs
  - Scope : general, structure ou service
  - Constant : User::ROLE_SECRETAIRE = 'secretaire'

4.2. Les 3 périmètres (scopes)

  Scope "general"
  ---------------
  - Portée sur toute l'organisation
  - Chef général : valide tout courrier, crée des entrants
  - Secrétaire général : même périmètre qu'un chef général mais sans
    validation
  - L'utilisateur n'est lié ni à une structure ni à un service spécifique
  - Constant : User::SCOPE_GENERAL = 'general'

  Scope "structure"
  -----------------
  - Portée sur une direction entière
  - Chef structure : gère le courrier de sa structure, valide, transmet
    (uniquement aux services de sa structure)
  - Secrétaire structure : travaille pour une structure
  - L'utilisateur est lié à une structure (structure_id renseigné)
  - Ne peut PAS être lié à un service (service_id = null)
  - Constant : User::SCOPE_STRUCTURE = 'structure'

  Scope "service"
  ---------------
  - Portée sur un département spécifique
  - Chef service : gère le courrier de son service uniquement
  - Secrétaire service : travaille pour un service
  - L'utilisateur est lié à un service (service_id renseigné)
  - La structure est déduite via le service parent
  - Constant : User::SCOPE_SERVICE = 'service'

4.3. Matrice des permissions par rôle/scope

  +----------------------+--------+--------+--------+--------+---------+
  | Action               | Admin  | Chef   | Chef   | Secr.  | Secr.   |
  |                      |        | Général| Service| Général| Service |
  +----------------------+--------+--------+--------+--------+---------+
  | Consulter utilisat.  | Oui    | Oui    | Oui    | Non    | Non     |
  | Créer utilisateur    | Oui    | Oui    | Son    | Non    | Non     |
  |                      |        |        | service|        |        |
  | Modifier utilisateur | Oui    | Oui    | Son    | Non    | Non     |
  |                      |        |        | service|        |        |
  | Supprimer utilisateur| Oui    | Non*   | Non*   | Non    | Non     |
  | Créer courrier       | Oui    | Oui    | Oui    | Oui    | Oui     |
  | Créer courrier entr. | Oui    | Oui    | Non    | Non    | Non     |
  | Valider courrier     | Oui    | Oui    | Service| Non    | Non     |
  | Transmettre courrier | Oui    | Oui    | Service| Non**  | Non**   |
  | Archiver courrier    | Oui    | Oui    | Oui    | Même   | Même    |
  |                      |        |        |        |service | service |
  +----------------------+--------+--------+--------+--------+---------+

  * La suppression d'un utilisateur est réservée à l'admin uniquement
    (vérifié dans UserPolicy::delete : l'admin ne peut pas se supprimer
    lui-même)
  ** Le secrétaire peut initier la transmission, mais elle passe en mode
     "demande de validation" (transmission_demandee = true)


5. HIERARCHIE DE GESTION DES UTILISATEURS
--------------------------------------------------------------------------------

La méthode User::peutGererUtilisateur(User $autre) définit qui peut gérer
qui dans l'organisation :

  Administrateur :
    -> Peut gérer TOUS les utilisateurs
    -> Constant : estAdmin() = true

  Chef général :
    -> Peut gérer TOUS les utilisateurs (sauf lui-même pour la suppression)
    -> Condition : estChef() && role_scope === 'general'

  Chef structure :
    -> Peut gérer uniquement les utilisateurs de SA structure
    -> L'utilisateur cible doit avoir structure_id === ma structure_id
    -> L'utilisateur cible ne doit pas être admin ni chef général
    -> Condition : estChefStructure() && structure_id !== null

  Chef service :
    -> Peut gérer uniquement les utilisateurs de SON service
    -> L'utilisateur cible doit avoir service_id === mon service_id
    -> L'utilisateur cible ne doit pas être admin
    -> Condition : peutGererUtilisateursDuService() = true

  Secrétaire (tout scope) :
    -> NE PEUT PAS gérer les utilisateurs
    -> Condition : estSecretaire()


6. REGLES DE VALIDATION DES FORMULAIRES
--------------------------------------------------------------------------------

6.1. Création (StoreUserRequest)
  - nom          : requis, chaîne, max 255 caractères
  - prenom       : requis, chaîne, max 255 caractères
  - email        : requis, email valide, max 255, unique dans users.email
  - password     : requis, confirmation requise, règles par défaut
                   (Laravel Password::defaults())
  - actif        : optionnel, booléen
  - role         : requis, doit être 'admin', 'chef' ou 'secretaire'
  - role_scope   : optionnel, doit être 'general', 'structure' ou 'service'
  - service_id   : optionnel, doit référencer un enregistrement existant
                   dans la table services
  - structure_id : optionnel, doit référencer un enregistrement existant
                   dans la table structures
  - niveau_confidentialite_id : optionnel, doit référencer un enregistrement
                   existant dans la table niveau_confidentialites

6.2. Modification (UpdateUserRequest)
  - Mêmes règles que la création, mais tous les champs sont optionnels
    ("sometimes")
  - email : unique dans users.email à l'exception de l'utilisateur en cours
    de modification
  - password : nullable (si vide, le mot de passe n'est pas modifié)

6.3. Règles métier côté frontend (Users.jsx)
  - Chef scope général : ne doit avoir ni structure_id ni service_id
    (forceNoStructure = true)
  - Chef scope structure : ne doit PAS avoir de service_id
    (forceNoService = true), mais DOIT avoir une structure
  - Secrétaire scope structure : DOIT avoir une structure
  - Secrétaire scope service : DOIT avoir une structure ET un service
  - Chef scope service : DOIT avoir une structure ET un service


7. CONTROLES D'ACCES DETAILLES
--------------------------------------------------------------------------------

7.1. UserPolicy

  Méthode       Condition d'accès
  ------------- ------------------------------------------------
  viewAny()     L'utilisateur courant peut consulter les
                utilisateurs (peutConsulterUtilisateurs())
                -> admin ou chef (tout scope) = OUI
                -> secretaire = NON

  view()        L'utilisateur courant peut voir le profil d'un
                utilisateur spécifique.
                -> peutGererUtilisateur($model) OU
                -> son propre profil ($user->id === $model->id)

  create()      Identique à viewAny()
                -> admin ou chef = OUI

  update()      L'utilisateur courant peut modifier un utilisateur.
                -> peutGererUtilisateur($model)
                (voir section 5 pour la logique hiérarchique)

  delete()      L'utilisateur courant peut supprimer un utilisateur.
                -> $user->id !== $model->id (pas d'auto-suppression)
                -> peutGererUtilisateur($model)
                (en pratique, réservé à admin vu que seul admin
                gère les chefs généraux)

7.2. Contrôleur UserController

  index()   GET /api/utilisateurs
    - Vérifie : $actor->can('viewAny', User::class)
    - Filtre les résultats selon le rôle de l'acteur :
      * Admin / Chef général -> voit TOUS les utilisateurs
      * Chef structure -> voit uniquement les utilisateurs de sa structure
      * Chef service -> voit uniquement les utilisateurs de son service
      * Secrétaire -> refusé (403)
    - Limite les données de référence (meta) :
      * Services listés = seulement ceux du périmètre de l'acteur
      * Structures listées = seulement celles du périmètre de l'acteur

  store()   POST /api/utilisateurs
    - Vérifie : $actor->can('create', User::class)
    - Vérifie canAssignRoleAndOrg() :
      * Admin / Chef général -> peut attribuer n'importe quel rôle
      * Chef structure -> peut créer uniquement si le role !== 'admin'
        ET structure_id === sa structure
      * Chef service -> peut créer uniquement si le role !== 'admin'
        ET service_id === son service

  update()  PATCH /api/utilisateurs/{id}
    - Vérifie : $actor->can('update', $user)
    - Vérifie canAssignRoleAndOrg() avec les nouvelles données
    - Si password est vide, il n'est pas modifié

  destroy() DELETE /api/utilisateurs/{id}
    - Vérifie : $actor->can('delete', $user)
    - L'admin ne peut pas se supprimer lui-même
    - Suppression physique (hard delete) de l'enregistrement


8. INTERFACE FRONTEND (Users.jsx)
--------------------------------------------------------------------------------

8.1. Page "Utilisateurs"
  - Tableau listant tous les utilisateurs avec :
    * Nom, prénom, email
    * Rôle avec badge coloré :
      - Administrateur (violet)
      - Chef (ambre)
      - Secrétaire (gris)
    * Périmètre avec badge coloré :
      - Général (émeraude)
      - Structure (bleu ciel)
      - Service (gris)
    * Structure et service d'affectation
    * Boutons d'action (Modifier, Supprimer) avec état disabled
      selon les permissions
  - Barre de recherche (filtre par nom, prénom, email)
  - Pagination (15 utilisateurs par page)
  - Cache localStorage (TTL : 60 secondes)

8.2. Modal de création / modification
  Champs du formulaire :
    - Nom (requis)
    - Prénom (requis)
    - Email (requis, validation format email)
    - Mot de passe (requis à la création, optionnel à la modification)
    - Confirmation du mot de passe
    - Rôle (select) :
      * Secrétaire (par défaut)
      * Chef
      * Administrateur (visible uniquement si l'acteur peut gérer
        les admins)
    - Périmètre (select) :
      * Service (par défaut)
      * Structure
      * Général
    - Structure (select, lié au périmètre) :
      * Options filtrées selon le périmètre de l'acteur
      * Désactivé si le scope est "général" (forceNoStructure)
    - Service (select, lié à la structure sélectionnée) :
      * Options filtrées par structure sélectionnée
      * Désactivé si scope "structure" ou "général" (forceNoService)
  
  Logique dynamique du formulaire :
    - Si rôle = "admin" -> scope forcé à "général", structure et
      service désactivés
    - Si rôle = "chef" + scope "général" -> pas de structure, pas de
      service
    - Si rôle = "chef" + scope "structure" -> structure requise,
      pas de service
    - Si rôle = "chef" + scope "service" -> structure + service requis
    - Si rôle = "secrétaire" + scope "structure" -> structure requise
    - Si rôle = "secrétaire" + scope "service" -> structure + service requis

8.3. Format des données renvoyées par l'API
  Chaque utilisateur est sérialisé avec :
    id, nom, prenom, nom_complet, email, role, role_scope, actif,
    service_id, service { id, libelle, structure_id },
    structure_id, structure { id, libelle },
    niveau_confidentialite_id,
    niveau_confidentialite { id, libelle, rang },
    peut_modifier (booléen calculé),
    peut_supprimer (booléen calculé)

8.4. Métadonnées (meta) renvoyées par l'API
  - roles : ['admin', 'chef', 'secretaire']
  - role_scopes : ['general', 'structure', 'service']
  - services : liste des services accessibles à l'acteur
  - structures : liste des structures accessibles à l'acteur
  - niveaux_confidentialite : tous les niveaux avec id, libelle, rang
  - peut_creer : booléen indiquant si l'acteur peut créer


9. REGLES METIER IMPORTANTES
--------------------------------------------------------------------------------

1. Auto-suppression interdite : un administrateur ne peut pas supprimer
   son propre compte (UserPolicy::delete)

2. Hiérarchie stricte : un chef de service ne peut créer que des
   utilisateurs de son service. Il ne peut pas créer d'administrateur.

3. Cohérence rôle/scope : 
   - Un admin a toujours le scope "general"
   - Un chef scope "general" n'a ni structure ni service
   - Un chef scope "structure" n'a pas de service

4. Visibilité des listes : un chef ne voit que les utilisateurs de son
   périmètre (service ou structure)

5. Confidentialité : le niveau de confidentialité d'un utilisateur
   détermine les courriers qu'il peut voir. Un utilisateur voit
   uniquement les courriers dont le rang de confidentialité est <=
   au sien.

6. Un utilisateur désactivé (actif = false) ne peut pas se connecter
   mais ses données historiques (courriers créés, validés, etc.)
   restent accessibles.

7. Cache : la liste des utilisateurs est mise en cache côté frontend
   pendant 60 secondes et est invalidée après chaque création,
   modification ou suppression.

11. ENDPOINTS API
--------------------------------------------------------------------------------

  Méthode  URL                         Description
  -------  --------------------------  --------------------------------
  GET      /api/utilisateurs           Liste paginée des utilisateurs
                                       + métadonnées (rôles, scopes,
                                       services, structures, niveaux
                                       de confidentialité)

  POST     /api/utilisateurs           Création d'un utilisateur
                                       Body : nom, prenom, email,
                                       password, password_confirmation,
                                       role, role_scope, service_id,
                                       structure_id,
                                       niveau_confidentialite_id, actif

  PATCH    /api/utilisateurs/{id}      Modification d'un utilisateur
                                       Body : (mêmes champs, tous
                                       optionnels sauf mot de passe
                                       qui devient nullable)

  DELETE   /api/utilisateurs/{id}      Suppression d'un utilisateur
                                       (admin uniquement, pas
                                       d'auto-suppression)
